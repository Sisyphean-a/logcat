## 1. 项目简介

当前仓库已经落地的现状是：桌面端启动后检测本机 `adb`、展示设备列表、选择一台 `device` 状态设备并实时展示 `chromium + [H5]` 日志；同时支持暂停/恢复、清空视图、基础搜索、复制、详情查看、自动跟随，以及包范围、前台 App、包/进程绑定和 PID 自动重绑。

项目仍处在从零到一阶段。高级查询、导出、多会话、Wi‑Fi ADB、崩溃增强识别和插件扩展仍只在 roadmap 里规划。

## 2. 核心概念 / 术语表

- **ADB 安装**：当前机器可执行的 `adb` 路径与版本信息
- **设备快照**：一次 `adb devices -l` 解析结果
- **包范围**：用户包 / 系统包 / 全部包的列表加载范围
- **进程快照**：某个包当前正在运行的相关进程集合，只保留 `PID` 和 `processName`
- **单活跃会话**：同一时刻只有一个 logcat reader 在运行；切换设备时旧会话必须先被取消
- **会话绑定**：当前日志会话锁定的设备、包、进程和活动 PID 集
- **PID 重绑**：目标包或进程的 PID 集变化后，取消旧会话并用新 PID 集重启
- **H5 预设**：命令层只看 `chromium`，本地层再过滤消息中包含 `[H5]` 的条目
- **可见日志**：当前主列表实际渲染的日志集合；与暂停期间的后台缓冲分离
- **暂停缓冲**：用户点击暂停后继续采集、但暂不追加到 UI 的日志集合
- **当前匹配**：搜索结果里当前聚焦的那一条，会同步选中行与详情面板
- **自动跟随**：主列表保持贴底；只有用户主动离开尾部或显式跳转时才关闭

## 4. 关键架构决定

- 当前先维持单窗口、单活跃会话，不在首个闭环里引入多窗口、多 Tab 或多会话。
- 所有错误直接上浮到状态区，不做静默回退。
- 当前输入格式固定为 `threadtime`，命令层先收窄到 `chromium`，业务层再过滤 `[H5]`，包 / 进程过滤继续在本地按 PID 集执行。
- 包名、前台包和进程发现统一留在 `adb-runtime`，不把 adb 原始文本上抛到 controller 或 UI。
- 暂停/恢复、搜索、清空视图、包/进程绑定和 PID 重绑都由 `app.Controller` 维护，不把这些 UI 控制回推到底层 adb 子进程。
- “前台 App”只做显式一次性选择，不做持续自动前台跟随。
- 自动跟随完全依赖 Gio 列表位置状态，不依赖 adb 或 session 层额外滚动协议。

## 5. 已知约束 / 硬边界

- 当前只能启动 `device` 状态设备；`offline` / `unauthorized` / `no permissions` 都会被阻断。
- 当前默认先进入设备级 H5 视图；只有用户显式选择包或进程后才收窄到绑定 PID 集。
- 当前相关进程的判定规则固定为“包名完全相等或 `package:` 前缀”。
- 当前不解析 App label、图标、版本号或 APK 元数据。
- 当前没有高级查询、持久化、导出、多窗口、多设备并行和 Wi‑Fi ADB 能力。
- 当前解析失败不会吞掉现场，原始输入会写进错误消息。
