# 直接查看并筛查 Android WebView 的 H5 日志

## 用户故事

- 作为一个在 Android 手机上调试 H5 页面的前端，我希望在电脑上直接看到 WebView 打出来的 H5 日志，而不是在海量原生日志里手动找。
- 作为一个调试宿主与 H5 交互问题的人，我希望先锁定 `chromium` 和 `[H5]` 这类日志，再决定下一步排查，而不是先学一串命令行参数。
- 作为一个还不知道宿主包名或 PID 的调试者，我希望先看用户包列表、前台 App 和相关进程，再把日志锁到目标包或目标进程。
- 作为一个没有装 Android Studio 的开发或测试，我希望只靠桌面工具就能开始看 H5 日志，而不是为了看几行 log 先打开整套 IDE。
- 作为一个正在排查 H5 问题的人，我希望在同一个桌面窗口里直接暂停、搜索、复制和查看当前日志细节，而不是把日志再倒回命令行二次处理。
- 作为一个长时间盯着日志流的人，我希望错误、请求、源码位置、链接和堆栈能在列表里自然跳出来，而不是整屏灰字靠肉眼硬扫。

## 为什么需要

H5 / WebView 调试最费时间的不是打日志，而是把日志从 Android 设备里捞出来并在电脑上持续筛查。原始 `adb logcat` 噪声很大，前端同学常常只关心 `chromium` 里那一小段 H5 输出，却要先在一大片原生日志里手工筛；即使已经找到目标流，后续还经常卡在“不知道宿主包名”“App 一重启 PID 变了”“日志混着多个进程”的低效往返里。

## 怎么解决

桌面工具启动后先帮用户确认本机 `adb` 和设备状态，再直接进入单设备日志查看链路，只把和 H5 调试最相关的那部分日志展示出来；进入列表后，用户可以在当前视图里暂停、恢复、搜索、复制并查看单条日志细节，也可以从用户包列表、前台 App 或相关进程里把日志进一步锁到目标包 / 目标进程，并在 App 重启导致 PID 变化时自动重绑。当前前端列表还会对错误、请求、URL、源码位置和堆栈前缀做轻量语义着色，降低长时间扫读成本。

## 边界

- 它当前解决的是“查看、当前视图筛查，并按包 / 进程锁定 H5 日志”这一步，不负责高级查询语言、导出、工作区、多设备并行、Wi‑Fi ADB 等后续操作。
- 它只面向 Android `adb` + WebView / `chromium` 日志，不覆盖 iOS 或其他日志协议。
- 使用前提是机器上已有可执行 `adb`，并且目标设备已经连上并处于可用状态。
- 它当前不解析 App label、图标、版本号或 APK 元数据，只关心包名、进程名和 PID。
- 它当前的语义着色只做前端展示，不做逐字符高亮、ANSI 解析、真实外链导航或后端 token 结构化。
